# Dataset Card: TrojanSpec-Bench v1.0

## Dataset summary

TrojanSpec-Bench is a benchmark of adversarially elicited formal
specifications. Each item pairs an honest natural-language requirement and its
faithful specification with a *trojaned* specification and a buggy witness
implementation that provably satisfies the trojan while violating the honest
spec. The benchmark targets three verifiers — Dafny, Lean 4, and Verus — and
four named attack patterns.

## Supported tasks

- **Trojan detection**: given `(nl_requirement, trojan_spec)`, predict that the
  spec is trojaned.
- **False-positive control**: given `(nl_requirement, original_spec)`, predict
  that the spec is clean.
- **Attack-pattern classification**: identify which of the four patterns a
  trojan uses.

## Languages

- Specification languages: Dafny, Lean 4, Verus.
- Natural language: English.

## Dataset structure

One `Triple` per record (see [`schema.md`](schema.md)). Splits are by
specification language: `dafny`, `lean`, `verus`. Difficulty stratification
(`easy / medium / hard`) is a field, not a split.

## Source data

Source problems are drawn from public verification benchmarks: FVAPPS,
DafnyBench, DafnyComp, the Vericoding benchmark, Verina, CloverBench, MBPP-DFY,
and HumanEval, plus cryptographic anchors derived from the public `libcrux`
codebase. Each source retains its original license; the `source_benchmark` and
`source_problem_hash` fields preserve provenance.

## Annotations

Trojans are machine-generated by adversarial LLM elicitors, then **reviewed by
a human** (`reviewed_by`, `review_passed`, `review_notes`) and **validated by a
real verifier** before admission. Only triples that pass both gates are
released.

## Considerations for using the data

This dataset has **dual-use** characteristics: it catalogues techniques for
making formal specifications silently wrong. It is released to advance
*defensive* research (specification auditing, AI control). Cryptographic
anchors reference only **publicly disclosed** weaknesses; see
[`disclosure_policy.md`](disclosure_policy.md). Do not use the trojan patterns
to introduce undisclosed weaknesses into production verification artefacts.

## Ethical considerations

- No undisclosed vulnerabilities are included.
- Trojan witnesses for cryptographic primitives map to bugs that already have
  public write-ups (IACR ePrint 2026/192, 2026/670).
- Maintainers of any newly surfaced bug class are notified under a 90-day
  coordinated-disclosure window before the corresponding entry is released.
